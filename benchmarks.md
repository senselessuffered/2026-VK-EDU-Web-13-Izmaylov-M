# Результаты нагрузочного тестирования (ДЗ 7)

Утилита: `ab` (Apache Benchmark).
Параметры: `-n 1000 -c 10` — 1000 запросов, по 10 одновременно.
Воспроизводится скриптом `run_bench.sh`, полные отчёты — в `bench/ab_*.txt`.

Сервисы:
- **nginx** — `:8080` (раздаёт статику с диска + `proxy_cache` для динамики);
- **gunicorn Django** — `:8000` (2 воркера, debug-toolbar выключен);
- **gunicorn simple wsgi** — `:8081` (голое WSGI-приложение без Django).

Команды:

```
ab -n 1000 -c 10 http://localhost:8080/sample.html         # 1
ab -n 1000 -c 10 http://localhost:8000/static/sample.html  # 2
ab -n 1000 -c 10 http://localhost:8081/                    # 3
ab -n 1000 -c 10 http://localhost:8080/nocache/            # 4
ab -n 1000 -c 10 http://localhost:8080/                    # 5 (после прогрева)
ab -n 1000 -c 10 http://localhost:8000/                    # 6
```

## Сводная таблица

| # | Что меряем | Запросов/сек | Время ответа |
|---|------------|-------------:|-------------:|
| 1 | Статика через **nginx** | **13 573** | 0.74 мс |
| 2 | Статика через **Django/gunicorn** | 2 755 | 3.63 мс |
| 3 | Динамика напрямую через gunicorn (simple wsgi, без Django) | 13 756 | 0.73 мс |
| 4 | Динамика через nginx → Django **без кеша** | 341 | 29.4 мс |
| 5 | Динамика через nginx → Django **с кешем** | **9 484** | 1.05 мс |
| 6 | Динамика напрямую через Django (без nginx) | 359 | 27.8 мс |

## Что из этого видно

### Статику быстрее отдаёт nginx, а не Django

Один и тот же файл `sample.html`:
- nginx — **13 573** запросов/сек;
- Django/gunicorn — **2 755** запросов/сек.

**nginx быстрее примерно в 5 раз.**

Почему: nginx — это программа на C, заточенная под одно — найти файл на диске и
отдать его в сокет (через системный вызов `sendfile`, почти без лишнего
копирования). Django же на каждый запрос поднимает Python-машинерию: проходит по
списку middleware, ищет файл через `staticfiles`-финдеры, собирает HTTP-ответ.
Для куска текста это лишняя работа.

### Каждый «живой» запрос в Django — дорогой (4 и 6)

Главная страница, которая реально считается в Django (ходит в БД за вопросами):
- напрямую — **359** запросов/сек;
- через nginx без кеша — **341** запросов/сек.

Цифры почти одинаковые → **узкое место не в nginx, а в самом Django + БД**.
nginx как прокси добавляет доли миллисекунды. Каждый такой запрос — это рендер
шаблона плюс запросы в PostgreSQL, отсюда ~28 мс на ответ.

### Кеш в nginx ускоряет динамику в ~28 раз

Та же главная страница:
- без кеша (каждый раз считает Django) — **341** запросов/сек;
- с `proxy_cache` (готовый ответ лежит у nginx) — **9 484** запросов/сек.

Почему: с кешем nginx запоминает первый ответ Django и следующую минуту
(`proxy_cache_valid 200 1m`) отдаёт эту копию сам, **вообще не дёргая Django и
БД**. Тяжёлый запрос (рендер + поход в базу) превращается в дешёвую отдай из
памяти. Кеш чуть медленнее чистой статики (9 484 против 13 573) только потому,
что страница большая (~40 КБ против 199 байт) — её дольше гнать по сети.

### Дело не в Python

Голое WSGI-приложение на том же gunicorn — **13 756** запросов/сек, а Django —
**341**. Сам Python и gunicorn быстрые; тормозит не язык, а то, что Django на
каждый запрос делает много работы (middleware, ORM, шаблоны, поход в БД).

## Главный вывод одним абзацем

Чем меньше работы на один запрос — тем быстрее. **nginx** почти не работает
(отдал готовый файл из памяти/диска) → десятки тысяч запросов в секунду.
**Django** работает много (шаблоны + БД) → сотни запросов в секунду. Поэтому
статику отдаём через nginx, а для динамики, где можно, ставим кеш: тогда Django
считает ответ один раз, а nginx раздаёт его копию всем остальным.
